# Poker Game State Protocol (Layer 2)

**Date:** 2026-04-13
**Status:** Draft
**Version:** 1.0

---

## Overview

This document defines the **Poker Game State Protocol** — the poker-specific content that travels inside the game-agnostic Transport Protocol (Layer 1). Layer 1 handles connection, authentication, turn-taking, timing, and error delivery. This document defines the structure and semantics of the game-specific payload fields that Layer 1 carries opaquely.

The current scope is **heads-up No-Limit Hold'em (NLHE)**. The protocol is designed to extend to multi-player tables and other variants without breaking changes.

---

## Layer 1 / Layer 2 Boundary

Layer 1 messages contain game-specific payload fields. This document defines the content of those fields for poker:

| Layer 1 Message        | Payload Field        | Layer 2 Definition                                      |
|------------------------|----------------------|---------------------------------------------------------|
| `match_start`          | `game_config`        | Match configuration (blinds, stacks, hand count)        |
| `round_start`          | `state`              | Hand start info (hand number, dealer, hole cards)       |
| `turn_request`         | `state`              | Current game state (phase, board, pot, stacks, actions)  |
| `turn_request`         | `valid_actions`      | Legal action strings from the poker action vocabulary   |
| `turn_action`          | `action`             | Chosen action string                                    |
| `turn_action`          | `params`             | Action parameters (e.g., raise amount)                  |
| `turn_result`          | `details`            | What happened (action, seat, amount)                    |
| `phase_change`         | `state`              | New board state after community cards dealt              |
| `round_result`         | `result`             | Hand outcome (winner, pot, payouts, showdown, history)  |

Layer 1 fields such as `server_ts`, `match_id`, `round_id`, `seq`, and timeout metadata are not defined here. They are present on every message and handled entirely by Layer 1. Note that `round_id` (a UUID) appears on `round_start` and `round_result` messages to correlate the start and end of each hand.

## 1. Card Notation

All cards use a two-character string: **rank** followed by **suit**.

### Ranks

| Character | Rank  |
|-----------|-------|
| `2`       | Two   |
| `3`       | Three |
| `4`       | Four  |
| `5`       | Five  |
| `6`       | Six   |
| `7`       | Seven |
| `8`       | Eight |
| `9`       | Nine  |
| `T`       | Ten   |
| `J`       | Jack  |
| `Q`       | Queen |
| `K`       | King  |
| `A`       | Ace   |

### Suits

| Character | Suit     |
|-----------|----------|
| `c`       | Clubs    |
| `d`       | Diamonds |
| `h`       | Hearts   |
| `s`       | Spades   |

### Examples

- `Ah` = Ace of hearts
- `Td` = Ten of diamonds
- `2c` = Two of clubs
- `Ks` = King of spades

A card string always matches the regex `^[2-9TJQKA][cdhs]$`.

## 2. Action Vocabulary

Poker defines five **player action** strings. These populate Layer 1's `valid_actions` array and are used in `turn_action.action`.

| Action    | Meaning                              | Requires `params`? |
|-----------|--------------------------------------|---------------------|
| `fold`    | Surrender the hand                   | No                  |
| `check`   | Pass when no bet to call             | No                  |
| `call`    | Match the current bet                | No                  |
| `raise`   | Increase the bet                     | Yes (`amount`)      |
| `all_in`  | Bet all remaining chips              | No                  |

`all_in` is a distinct action rather than a `raise` at max. This simplifies bot logic and removes ambiguity when a player's remaining stack is less than the minimum raise.

### Synthetic Actions (Server-Generated)

The following actions are **synthetic** — generated by the server, not submitted by players. They appear in `action_history` to provide a complete audit trail but are never included in `valid_actions`.

| Action             | Meaning                                    |
|--------------------|--------------------------------------------|
| `post_small_blind` | Small blind posted at the start of a hand  |
| `post_big_blind`   | Big blind posted at the start of a hand    |
| `post_ante`        | Ante posted at the start of a hand         |

Synthetic actions always appear at the **start** of `action_history` for each hand, before any player actions. When antes are configured (`ante > 0` in `game_config`), ante entries appear after blind entries. Example opening sequence:

```json
[
  {"seat": 0, "action": "post_small_blind", "amount": 5, "phase": "preflop", "is_timeout": false},
  {"seat": 1, "action": "post_big_blind", "amount": 10, "phase": "preflop", "is_timeout": false},
  {"seat": 0, "action": "post_ante", "amount": 5, "phase": "preflop", "is_timeout": false},
  {"seat": 1, "action": "post_ante", "amount": 5, "phase": "preflop", "is_timeout": false}
]
```

## 3. Payload Schemas

### 3.1 Game Config (`match_start.game_config`)

Sent once at the start of a match. Defines the rules for all hands in the match.

```json
{
  "variant": "nlhe",
  "starting_stack": 1000,
  "small_blind": 5,
  "big_blind": 10,
  "ante": 0,
  "total_hands": 0
}
```

| Field            | Type    | Required | Description                                                    |
|------------------|---------|----------|----------------------------------------------------------------|
| `variant`        | string  | Yes      | Game variant. Currently always `"nlhe"`.                       |
| `starting_stack` | integer | Yes      | Chips each player starts with. Must be > 0.                   |
| `small_blind`    | integer | Yes      | Small blind amount. Must be > 0.                               |
| `big_blind`      | integer | Yes      | Big blind amount. Must be >= `small_blind`.                    |
| `ante`           | integer | No       | Per-player ante posted each hand. Default `0`.                 |
| `total_hands`    | integer | Yes      | Number of hands to play. `0` = elimination (play until bust).  |

### 3.2 Round Start State (`round_start.state`)

Sent at the beginning of each hand, before any actions.

```json
{
  "hand_number": 1,
  "dealer_seat": 0,
  "your_hole_cards": ["Ah", "Kd"],
  "stacks": [1000, 990],
  "deck_commitment": ""
}
```

| Field              | Type            | Required | Description                                             |
|--------------------|-----------------|----------|---------------------------------------------------------|
| `hand_number`      | integer         | Yes      | 1-indexed hand number within the match.                 |
| `dealer_seat`      | integer         | Yes      | Seat index of the dealer (button). 0-indexed.           |
| `your_hole_cards`  | array of string | Yes      | Exactly 2 card strings dealt to the receiving player.   |
| `stacks`           | array of integer| Yes      | Chip stacks indexed by seat, before blinds are posted.  |
| `deck_commitment`  | string          | Yes      | `SHA-256(deck_seed \|\| deck_order)` where `deck_order` is the full 52-card sequence as a joined string. Empty string `""` if RNG verification is not enabled for this competition. See Section 6. |

### 3.3 Turn Request State (`turn_request.state`)

The core decision payload. Contains everything a bot needs to choose an action.

```json
{
  "hand_number": 1,
  "phase": "flop",
  "board": ["Ts", "7h", "2d"],
  "your_hole_cards": ["Ah", "Kd"],
  "pot": 40,
  "your_stack": 960,
  "opponent_stacks": [940],
  "to_call": 20,
  "min_raise": 40,
  "max_raise": 960,
  "action_history": [
    {"seat": 0, "action": "post_small_blind", "amount": 5, "phase": "preflop", "is_timeout": false},
    {"seat": 1, "action": "post_big_blind", "amount": 10, "phase": "preflop", "is_timeout": false},
    {"seat": 0, "action": "raise", "amount": 30, "phase": "preflop", "is_timeout": false},
    {"seat": 1, "action": "call", "amount": 30, "phase": "preflop", "is_timeout": false}
  ]
}
```

| Field             | Type                   | Required | Description                                                                                     |
|-------------------|------------------------|----------|-------------------------------------------------------------------------------------------------|
| `hand_number`     | integer                | Yes      | Current hand number (1-indexed).                                                                |
| `phase`           | string                 | Yes      | Current betting phase: `"preflop"`, `"flop"`, `"turn"`, or `"river"`.                           |
| `board`           | array of string        | Yes      | Community cards dealt so far. Empty `[]` for preflop. 3, 4, or 5 cards for later phases.        |
| `your_hole_cards` | array of string        | Yes      | The 2 hole cards dealt to the acting player.                                                    |
| `pot`             | integer                | Yes      | Total chips in the pot, including bets in the current round.                                    |
| `your_stack`      | integer                | Yes      | The acting player's remaining stack (chips not yet in the pot).                                 |
| `opponent_stacks` | array of integer       | Yes      | Other players' remaining stacks, ordered by seat (excluding the acting player's seat).          |
| `to_call`         | integer                | Yes      | Amount the acting player must add to call. `0` when checking is free.                           |
| `min_raise`       | integer                | Yes      | Minimum legal raise total. `0` if raising is not a valid action.                                |
| `max_raise`       | integer                | Yes      | Maximum legal raise total (the acting player's effective stack). `0` if raising is not valid.   |
| `action_history`  | array of ActionEntry   | Yes      | All actions taken this hand so far, in chronological order. Includes synthetic blind/ante entries at the start. |

**Raise semantics:** `min_raise` and `max_raise` represent the **total bet size**, not the incremental amount above the current bet. A bot submitting `raise` with `amount: 60` means the total bet is 60, not 60 on top of the existing bet.

### 3.4 Action Entry Schema

Used in `action_history` arrays throughout the protocol. The `action_history` in `round_result.result` is the **canonical source** for the complete action history of a hand. Layer 1 does not carry action history — it lives exclusively in Layer 2.

```json
{
  "seat": 0,
  "action": "raise",
  "amount": 30,
  "phase": "preflop",
  "is_timeout": false
}
```

| Field        | Type    | Required | Description                                                                |
|--------------|---------|----------|----------------------------------------------------------------------------|
| `seat`       | integer | Yes      | 0-indexed seat of the player who acted.                                    |
| `action`     | string  | Yes      | One of the action vocabulary strings (see Section 2).                      |
| `amount`     | integer | Yes      | Chips committed for this action. `0` for `fold` and `check`.              |
| `phase`      | string  | Yes      | The betting phase during which the action occurred.                        |
| `is_timeout` | boolean | Yes      | `true` if this action was auto-applied by the server due to timeout. Makes timeout-forced actions distinguishable without cross-referencing Layer 1 timeout messages. |

### 3.5 Turn Action Params (`turn_action.params`)

Parameters accompanying the chosen action in `turn_action.action`.

| Action    | `params` value                         | Constraint                                         |
|-----------|----------------------------------------|----------------------------------------------------|
| `fold`    | `{}` or omitted                        | None.                                              |
| `check`   | `{}` or omitted                        | None.                                              |
| `call`    | `{}` or omitted                        | None.                                              |
| `raise`   | `{"amount": <integer>}`                | `min_raise <= amount <= max_raise`                 |
| `all_in`  | `{}` or omitted                        | None. Server uses the player's full remaining stack.|

**Example — raise to 60:**
```json
{
  "action": "raise",
  "params": {"amount": 60}
}
```

**Example — fold:**
```json
{
  "action": "fold",
  "params": {}
}
```

### 3.6 Turn Result Details (`turn_result.details`)

Broadcast to all participants after each action. Reveals what a player did (but not their hole cards).

```json
{
  "seat": 1,
  "action": "call",
  "amount": 20
}
```

| Field   | Type    | Required | Description                                              |
|---------|---------|----------|----------------------------------------------------------|
| `seat`  | integer | Yes      | Seat of the player who acted.                            |
| `action`| string  | Yes      | The action taken.                                        |
| `amount`| integer | Yes      | Chips committed. `0` for `fold` and `check`.             |

### 3.7 Phase Change State (`phase_change.state`)

Sent when the betting round completes and new community cards are dealt.

```json
{
  "phase": "turn",
  "board": ["Ts", "7h", "2d", "Qc"]
}
```

| Field   | Type            | Required | Description                                                         |
|---------|-----------------|----------|---------------------------------------------------------------------|
| `phase` | string          | Yes      | The new betting phase: `"flop"`, `"turn"`, or `"river"`.           |
| `board` | array of string | Yes      | All community cards including the newly dealt card(s).              |

Phase transitions and card counts:
- `preflop` -> `flop`: board goes from `[]` to 3 cards
- `flop` -> `turn`: board goes from 3 to 4 cards
- `turn` -> `river`: board goes from 4 to 5 cards

### 3.8 Round Result (`round_result.result`)

Sent at the conclusion of each hand. Provides the complete audit trail.

```json
{
  "hand_number": 1,
  "winner_seats": [0],
  "pot": 60,
  "payouts": [
    {"seat": 0, "amount": 60}
  ],
  "showdown": [
    {"seat": 0, "hole_cards": ["Ah", "Kd"], "hand_rank": "pair"},
    {"seat": 1, "hole_cards": ["Jc", "Tc"], "hand_rank": "high_card"}
  ],
  "action_history": [
    {"seat": 0, "action": "post_small_blind", "amount": 5, "phase": "preflop", "is_timeout": false},
    {"seat": 1, "action": "post_big_blind", "amount": 10, "phase": "preflop", "is_timeout": false},
    {"seat": 0, "action": "raise", "amount": 30, "phase": "preflop", "is_timeout": false},
    {"seat": 1, "action": "call", "amount": 30, "phase": "preflop", "is_timeout": false},
    {"seat": 0, "action": "check", "amount": 0, "phase": "flop", "is_timeout": false},
    {"seat": 1, "action": "check", "amount": 0, "phase": "flop", "is_timeout": false},
    {"seat": 0, "action": "check", "amount": 0, "phase": "turn", "is_timeout": false},
    {"seat": 1, "action": "check", "amount": 0, "phase": "turn", "is_timeout": false},
    {"seat": 0, "action": "check", "amount": 0, "phase": "river", "is_timeout": false},
    {"seat": 1, "action": "check", "amount": 0, "phase": "river", "is_timeout": false}
  ],
  "stacks": [1030, 970],
  "deck_commitment": "",
  "deck_reveal": null
}
```

| Field            | Type                   | Required | Description                                                                                 |
|------------------|------------------------|----------|---------------------------------------------------------------------------------------------|
| `hand_number`    | integer                | Yes      | The hand number that just completed.                                                        |
| `winner_seats`   | array of integer       | Yes      | Seat(s) that won. Multiple entries for split pots.                                          |
| `pot`            | integer                | Yes      | Total pot size for the hand.                                                                |
| `payouts`          | array of PayoutEntry   | Yes      | Array of `{"seat": <int>, "amount": <int>}` objects. All seats in `winner_seats` appear here. Consistent with other seat-indexed structures in the protocol. |
| `showdown`         | array of ShowdownEntry | Yes      | Players who reached showdown. Empty array `[]` if hand ended before showdown (e.g., fold).  |
| `action_history`   | array of ActionEntry   | Yes      | Complete ordered list of every action in the hand, including synthetic blind/ante entries. This is the **canonical source** for the hand's action history. Full audit trail. |
| `stacks`           | array of integer       | Yes      | Updated chip stacks after payouts, indexed by seat.                                         |
| `deck_commitment`  | string                 | Yes      | `SHA-256(deck_seed \|\| deck_order)` commitment from `round_start`. Empty string `""` if RNG verification is not enabled. See Section 6. |
| `deck_reveal`      | object or null         | Yes      | When RNG verification is enabled: `{"seed": "<hex>", "deck_order": ["Ah", "Kd", ...]}`. When not enabled: `null`. Participants verify: `SHA-256(seed \|\| join(deck_order)) == deck_commitment`. See Section 6. |

### 3.9 Showdown Entry Schema

```json
{
  "seat": 0,
  "hole_cards": ["Ah", "Kd"],
  "hand_rank": "pair"
}
```

| Field       | Type            | Required | Description                                             |
|-------------|-----------------|----------|---------------------------------------------------------|
| `seat`      | integer         | Yes      | Seat of the player at showdown.                         |
| `hole_cards`| array of string | Yes      | The player's 2 hole cards.                              |
| `hand_rank` | string          | Yes      | Winning hand classification (see hand rank values).     |

### Hand Rank Values

| Value              | Hand                |
|--------------------|---------------------|
| `high_card`        | High card           |
| `pair`             | One pair            |
| `two_pair`         | Two pair            |
| `three_of_a_kind`  | Three of a kind     |
| `straight`         | Straight            |
| `flush`            | Flush               |
| `full_house`       | Full house          |
| `four_of_a_kind`   | Four of a kind      |
| `straight_flush`   | Straight flush      |
| `royal_flush`      | Royal flush         |

## 5. Semantic Rules

### 5.1 Blind Posting

Blinds are posted automatically by the server before any turn requests. They are recorded in `action_history` as **synthetic entries** using the `post_small_blind` and `post_big_blind` actions. These entries always appear at the start of the action history, before any player actions. The pot in the first `turn_request` of a hand already includes posted blinds. Stack values already reflect blind deductions.

### 5.2 Ante Posting

When `ante > 0` in `game_config`, antes are posted automatically alongside blinds. They are recorded in `action_history` as synthetic `post_ante` entries, appearing after blind entries and before player actions. Pot and stack values reflect ante deductions.

### 5.3 Heads-Up Blind Structure

In heads-up (2-player) play:
- The **dealer** (button) posts the **small blind** and acts **first preflop**.
- The **non-dealer** posts the **big blind** and acts **first post-flop** (flop, turn, river).

### 5.4 Raise Sizing

- `min_raise` is the minimum legal **total bet**. In standard NLHE, the minimum raise is the previous raise increment plus the current bet to match.
- `max_raise` is the acting player's effective stack (the most they can put in).
- If a player's stack is less than `min_raise`, `raise` is not in `valid_actions`. The player may still go `all_in`.
- The `amount` in raise params is the **total bet size**, not the increment.

### 5.5 All-In

- `all_in` commits the player's entire remaining stack.
- It appears in `valid_actions` whenever the player has chips and is not already all-in.
- When a player goes all-in for less than a full raise, this does **not** reopen betting for a player who has already acted in this round.

### 5.6 Split Pots

When two or more players tie at showdown, `winner_seats` contains all tied seats and `payouts` contains an entry for each winning seat with the pot distributed equally (with any odd chip going to the player in earliest seat position).

### 5.7 Side Pots

In multi-player scenarios where players are all-in for different amounts, the server calculates main and side pots internally. The `round_result` reports net payouts per seat — the payout each seat receives, not the breakdown of individual side pots.

### 5.8 Timeout Behavior

Timeouts are handled by Layer 1. When a player times out, the server substitutes a default action: `check` if checking is free, otherwise `fold`. This substituted action flows through `turn_result` and `action_history` normally. The `is_timeout` field on the action entry is set to `true` for timeout-forced actions, making them distinguishable from player-chosen actions without cross-referencing Layer 1 timeout messages.

## 6. Regulatory and Integrity Fields

The protocol includes fields to support fair play verification:

| Feature                | Mechanism                                                                      |
|------------------------|--------------------------------------------------------------------------------|
| Complete action history | `round_result.action_history` contains every action by every player per hand, including synthetic blind/ante entries and `is_timeout` flags. |
| Timestamps             | Layer 1 `server_ts` on every message provides a tamper-evident timeline.      |
| Deck verification      | Commit-reveal scheme via `deck_commitment` and `deck_reveal` fields.          |
| Card retention          | Server retains all dealt cards (including mucked hands) for dispute resolution.|
| Deterministic replay   | Complete action history + deck reveal enables full hand reconstruction.        |

### 6.1 Deck Commit-Reveal Verification

The protocol defines a commit-reveal scheme for RNG verification. This allows participants to verify that the deck was not altered mid-hand.

**Commitment phase** (`round_start.state.deck_commitment`):

Before dealing, the server generates a random `deck_seed` and shuffles the deck to produce `deck_order` (the full 52-card sequence). The commitment is:

```
deck_commitment = SHA-256(deck_seed || join(deck_order))
```

Where `join(deck_order)` concatenates the 52 two-character card strings (e.g., `"AhKd2c7s..."`). The commitment is included in the `round_start` state sent to all participants.

**Reveal phase** (`round_result.result.deck_reveal`):

After the hand completes, the server reveals the seed and full deck order:

```json
{
  "seed": "a3f2b1c4d5e6f7...",
  "deck_order": ["Ah", "Kd", "2c", "7s", "..."]
}
```

**Verification**: Participants verify that `SHA-256(seed || join(deck_order)) == deck_commitment` from the corresponding `round_start`. They can also verify that the dealt cards (hole cards, board) match the expected positions in `deck_order`.

**v1 note**: v1 implementations MAY leave `deck_commitment` as an empty string `""` and `deck_reveal` as `null`. The schema is defined and ready for when RNG verification is enabled for a competition.

## 7. Versioning

This document describes **v1** of the Poker Game State Protocol. The version is communicated via Layer 1's connection handshake (outside this document's scope). Changes to this protocol follow these rules:

- **Additive changes** (new optional fields): minor version bump, backward compatible.
- **Breaking changes** (field removal, type changes, semantic changes): major version bump, requires client updates.
- **New variants** (e.g., Pot-Limit Omaha): new `variant` value in `game_config`, separate Layer 2 specification document.
